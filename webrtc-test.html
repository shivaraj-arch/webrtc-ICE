📊 DIAGNOSTIC INFO

⚠️ MULTI-DEVICE SETUP:
Do NOT use http://localhost:3000 on the second device!
Instead, find your server's local IP (like 192.168.1.5 or 10.0.0.50) and use http://:3000 on both devices.
Why: WebRTC needs real addresses to negotiate - loopback only works on the same machine.

✅ To test WebRTC:

  1. Same Machine: Open localhost:3000 in TWO browser tabs. Select different roles. Click START.
  2. Different Machines: Get your server's IP from terminal: ifconfig or ipconfig
  3. Device A: Go to http://YOUR_IP:3000, select "INITIATOR", click "START WEBRTC"
  4. Device B: Go to http://YOUR_IP:3000, select "RESPONDER", click "START WEBRTC"
  5. Watch logs. When both show ICE "connected", click "📊 Dump Connection Stats"
  6. If data channel opens (✅ OPEN), try "📤 SEND TEST AUDIO"

💡 STUN/TURN Servers Being Used:
STUN: Google (5 servers) - discovers public IP
TURN: Viagenie, OpenRelay (2 ports) - relay fallback

Candidate Types:
[HOST] = Local IP (LAN)
[SRFLX] = Public IP from STUN
[RELAY] = Proxied via TURN server

⚠️ Same Public IP in Dump?
If both peers show the same IP (e.g., 49.37.183.2), it means you're both on the same ISP/network. The different PORTS (e.g., :14278 vs :27584) identify each peer. This is normal and working correctly!

If connection fails: Check "📊 Dump Connection Stats" → if no "Working Pair", NAT is blocking UDP
